# AGENTS.md — podkop-wiki

This file is the operating manual for AI coding agents (Devin, Claude Code, Cursor, etc.) working in this repository. Read it fully before making any changes. Humans should read it too.

## 1. Project identity

- **Owner / fork status:** `BlindMaster24/podkop-wiki` is **not a GitHub fork**. It is an independent personal mirror of `itdoginfo/podkop-wiki`, the source of `podkop.net`.
- **Upstream:** `https://github.com/itdoginfo/podkop-wiki` (read-only).
- **License:** GPL-2.0-or-later (inherited from upstream). Preserve `LICENSE`.
- **Companion repo:** `BlindMaster24/podkop-ng` (the software the docs describe). When documenting features, source-of-truth for behavior is that repo, not user assumptions.
- **Scope of changes in this fork:** documentation that matches this fork of the software; in particular **accessibility-aware install/configuration guides** for users on screen readers or keyboard-only setups. Other docs work is welcome; there is no fixed scope beyond a11y emphasis.
- **Upstream relationship:** treat this as a **separate project**, not as a maintained downstream. There is no obligation to keep upstream-sync compatibility; do not constrain new work for the sake of easy upstream merges. Occasional cherry-picks from `itdoginfo/podkop-wiki` may still happen; **the AI agent — not the human owner — is the one expected to perform them when the owner asks**, using the workflow in §4. AGENTS.md keeps those commands ready for that reason.

## 2. What this repository builds

A static documentation site at `https://podkop.net/`, generated by **Hugo** using the **Hextra** theme (Hugo Modules), Go-managed.

Key facts:

- Static site generator: Hugo (the upstream CD pipeline pins `0.147.3`).
- Theme: `github.com/imfing/hextra` v0.9.7, imported via Hugo Modules (`hugo.yaml → module.imports`, `go.mod`).
- Default content language: Russian (`defaultContentLanguage: ru`).
- Markdown renderer: `goldmark` with `unsafe: true` (raw HTML allowed in Markdown).
- Analytics: Google Analytics ID `G-L8FLJF5S5R` is configured in `hugo.yaml` (upstream). Treat this as upstream's tracker, not ours; remove or replace it before publishing under our own domain (see §8).

## 4. Upstream sync workflow (run by the AI agent on demand)

This repo is developed as a separate project; nothing here syncs to upstream automatically or on a schedule. **This section is the procedure the AI agent runs when the owner asks for one or more upstream commits to be pulled in.** The owner is not expected to run these commands themselves.

Upstream is configured as a remote in fresh local clones:

```sh
git remote add upstream https://github.com/itdoginfo/podkop-wiki.git
git remote set-url --push upstream DISABLED
```

To pull upstream changes when asked:

```sh
git fetch upstream
git log upstream/main --oneline ^main
git cherry-pick <sha>...                   # selective (default mode)
# or, only if the owner explicitly asks for a bulk merge:
git merge --no-ff upstream/main
```

Default to selective cherry-picks of the specific commits the owner names. Use `merge --no-ff` only on explicit owner request. When resolving conflicts, accessibility-specific additions in this repo always win; do not pull in upstream Google Analytics changes (see §8). Never push to `upstream`.

## 5. Build, preview, deploy

### 5.1 Local preview

Hugo extended is **not required** for this site (the upstream `deploy.yml` keeps `extended: true` commented out). Standard Hugo works.

```sh
# install hugo (>=0.147.3 recommended; matches upstream CI)
# Debian/Ubuntu (snap):
sudo snap install hugo
# or download binary from https://github.com/gohugoio/hugo/releases

# fetch theme module
hugo mod get -u
hugo mod tidy

# preview at http://localhost:1313
hugo server -D
```

### 5.2 Production build

```sh
hugo
# output goes to ./public/
```

The build is hermetic: `./public/` should contain everything needed to serve the site. Do not commit `./public/`; it is build output.

### 5.3 Deployment

Upstream's `.github/workflows/deploy.yml` deploys via `rsync -a ./public/* /var/podkop/ --delete` on a `self-hosted` runner. **This fork does not have access to that runner or that machine.** Either:

1. Disable the workflow on this fork (preferred until a real deploy target exists), or
2. Replace it with a workflow targeting a publicly-hostable target (GitHub Pages, Cloudflare Pages, Netlify) and `runs-on: ubuntu-latest`.

Before any first deployment under our control, also do §8.

### 5.4 Editing content

- New page in an existing section: `hugo new docs/<section>/<page>.md` then edit.
- New section: create `content/docs/<section>/_index.md` plus children. Add an `images/` subfolder for screenshots that belong to that page. Reference images relative to the page: `![alt](images/foo.png)`.
- All Markdown files must include front matter (`title:` at minimum). The repo uses YAML front matter throughout — do not switch to TOML.
- Headings start at H2 inside a doc page (the page title is rendered from front matter as H1 by the theme).

## 6. Coding / authoring conventions (binding for AI agents)

These conventions apply to **every** change made by an AI agent in this repo.

1. **Language of repository artifacts: English everywhere.**
   - File names, branch names, commit messages, PR titles/bodies, AGENTS.md updates, comments in `layouts/`, `assets/`, build scripts — **English only**.
   - **Exception, deliberate:** user-facing **content** (`content/**.md`) is currently written in Russian (`defaultContentLanguage: ru`). New translations of existing pages should live under a sibling locale path (e.g. `content/en/...`) and be configured via Hugo's i18n; do not silently switch a page from Russian to English in place.

2. **Comments: avoid them.**
   - Templates (`layouts/**/*.html`), shortcodes, and config files default to **no comments**. Pick clear partial names instead.
   - When unavoidable (a non-obvious Hextra theme override quirk, a Goldmark behavior, a deployment-only constraint), comments are: English, ≤80 chars, describe the **why**, never the **what**.
   - Never narrate the diff in comments; that belongs in the commit message.
   - In Markdown content, do not add HTML comments unless required for theme behavior.

3. **Commit messages: Conventional Commits, in English, imperative mood.**
   - Format: `type(scope): subject`
     - `type` ∈ `feat | fix | docs | content | style | refactor | build | ci | chore | revert`
     - `scope` (optional) ∈ `content | layouts | assets | hugo | ci | upstream-sync | a11y` or a directory.
     - `subject`: imperative, no trailing period, ≤72 chars.
   - Body (optional, after a blank line): wrap at 72 chars, explain **why**.
   - Examples:
     - `content(install): document apk vs ipk decision tree`
     - `feat(layouts): add skip-to-content link in baseof`
     - `chore(upstream-sync): cherry-pick itdoginfo@d4e5f6a`
     - `ci: replace self-hosted deploy with cloudflare pages`
   - One logical change per commit. Never amend pushed commits. Never force-push to `main`.

4. **Branches.**
   - `main` is the default integrated branch.
   - Feature branches: `content/<short-kebab>`, `feat/<short-kebab>`, `fix/<short-kebab>`, `a11y/<short-kebab>`, `chore/<short-kebab>`.
   - Upstream syncs: `sync/upstream-<yyyy-mm-dd>`.

5. **PRs.**
   - Title in Conventional Commits format.
   - Description includes: motivation, what changed, screenshots of changed pages (`hugo server` and capture rendered output), confirmation that `hugo` builds without errors.

6. **Accessibility (binding for content and layout work).**
   - Every image must have meaningful `alt` text in the language of the surrounding content. Decorative-only images use empty `alt=""`.
   - Heading hierarchy must not skip levels (H2 → H3 → H4, never H2 → H4).
   - Tables must have header cells; complex tables must specify `scope`.
   - Color contrast on rendered pages must meet WCAG 2.1 AA (≥4.5:1 body text, ≥3:1 large text and UI). Verify after theme upgrades.
   - Custom layout overrides must include keyboard-reachable equivalents of any mouse-only interaction (no `:hover`-only menus, no click-to-toggle without keyboard).
   - Add and maintain a "skip to content" link in the base layout.
   - Do not rely on color, position, or icons alone to convey meaning.

7. **Theme (Hextra) updates.**
   - Bump theme version in `go.mod` only via `hugo mod get -u github.com/imfing/hextra@<version>` followed by `hugo mod tidy`.
   - Re-run `hugo` and verify all custom overrides in `layouts/` still apply (Hextra changes can break partials).
   - Run an a11y spot-check (NVDA/Orca/VoiceOver) on the home page, install page, and one settings page before merging a theme bump.

## 7. AI-agent runbook

Concrete protocol when an agent is asked to implement a change:

1. **Plan.** State the goal, list affected files (content vs layout vs config), list the validation steps.
2. **Read first.** Open every file you intend to touch and the closest neighbors. Read `hugo.yaml` and any related `_index.md` before editing children.
3. **Branch.** `git switch -c <type>/<short-kebab>` from up-to-date `main`.
4. **Edit.** Apply the minimal diff. No drive-by reformatting of unrelated content files.
5. **Validate locally:**
   - `hugo mod tidy` if dependencies changed.
   - `hugo` (full build) — must complete with zero errors. Warnings should be triaged.
   - `hugo server -D` and visit each affected page; capture screenshots for the PR.
   - For accessibility-tagged changes, also run `axe`/`Pa11y` against the local server and attach the report.
6. **Commit.** One Conventional Commit per logical change. English, imperative.
7. **PR.** Open against `main` with screenshots and build evidence.
8. **Wait for CI.** Do not declare done until CI is green.
9. **Sync (only when the owner asks).** When the owner names one or more commits from `itdoginfo/podkop-wiki` to pull in, open a `sync/upstream-<date>` branch, cherry-pick those commits (or `merge --no-ff` only on explicit owner request), run `hugo` to verify the site still builds, resolve conflicts (a11y-related conflicts always favor preserving accessibility work; drop upstream GA changes per §8), open PR. Do not initiate syncs on your own.

## 8. Things to fix or escalate before public deployment under our domain

The repository was authored to deploy `podkop.net` from `itdoginfo/podkop-wiki`. Before this fork is ever published under a domain we control, escalate or address:

- `hugo.yaml → baseURL` is `https://podkop.net/`. Change to our actual domain or to the pages URL of the deploy target.
- `hugo.yaml → params.editURL.base` points to `https://github.com/itdoginfo/podkop-wiki/edit/main/content/`. Update to our fork.
- `hugo.yaml → services.googleAnalytics.ID` belongs to upstream. Remove it (or replace with our own GA/Plausible/Umami ID) before any production deploy.
- `hugo.yaml → menu.main` links to `https://github.com/itdoginfo/podkop` and the upstream Telegram thread. Decide whether our docs link to upstream community resources or to our own.
- `.github/workflows/deploy.yml` targets a `self-hosted` runner with rsync access to `/var/podkop/`. Replace with a deploy target this fork can actually use, or disable the workflow.

These are deliberately not changed eagerly: until we have a deployment target, leaving the upstream values keeps the diff against upstream small. Open a tracking issue when the deploy target is decided.

## 9. Things to escalate

Stop and ask the human owner before doing any of the following:

- Switching SSG (Hugo → anything else).
- Switching theme (Hextra → anything else).
- Restructuring `content/docs/**` (mass rename or re-org).
- Adding telemetry beyond what §8 already covers.
- Touching `LICENSE`.
- Force-pushing to `main` or any shared branch.
- Auto-merging the upstream `main` without an explicit owner request (the agent must wait to be asked; see §4).
